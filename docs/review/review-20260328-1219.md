# Review Report

## 摘要

本次评审范围为单文档审查：`smartclaw/docs/SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md`。整体设计方向是合理的，核心判断“不要固定 workflow，也不要完全自由 agent，而是做动态规划 + 配置驱动约束”是对的，适合作为 SmartClaw 超级智能体框架的总方向。

当前主要问题不在大方向，而在若干关键运行时契约还不够明确：`step` 适用条件、`planner -> orchestrator` 交接结构、`artifact` 的最小标准化约束、治理规则的优先级、replanning 的终止边界。如果不在 Phase 1 开工前补清楚，这套设计容易在实现时重新退化成硬编码或出现运行时不稳定。以下问题可能未穷尽；本次仅基于单文档内容做审查，未做文档与代码的逐项一致性核验。

## 问题清单

### 高

1. 缺少最小“step 可执行契约”，会导致 planner 难以稳定判断何时可选、何时缺输入。
   - 证据：文档明确要求 Step Registry 仅保留最小字段，并刻意不引入 `applicable_when` 与复杂条件配置，但 planner 又需要判断“是否要补前置步骤”“是否还缺关键输入”“当前最合理的一组 todos”。见 [SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md](/Users/liubu/hx/hxWork/35.AI测试/claw/smartclaw/docs/SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md#L107), [SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md](/Users/liubu/hx/hxWork/35.AI测试/claw/smartclaw/docs/SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md#L239), [SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md](/Users/liubu/hx/hxWork/35.AI测试/claw/smartclaw/docs/SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md#L650)
   - 风险：一旦业务域增多，planner 只能依赖 prompt 推测 step 的适用性，后续很容易出现“选了一个实际上前置条件不满足的 step”或“漏补必要前置步骤”的问题，动态规划会变成 prompt heuristics。

2. `planner -> orchestrator` 的交接结构过薄，无法支撑稳定执行、审计和回放。
   - 证据：Planner 输出示例只有 `id / title / inputs / depends_on / parallelizable`，但 orchestrator 需要分派具体执行、控制重试、做审批和结果回传。文档没有定义 todo 实例如何绑定 artifact 输入、执行目标、skill/tool 解析结果、风险决策结果或 schema/version。见 [SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md](/Users/liubu/hx/hxWork/35.AI测试/claw/smartclaw/docs/SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md#L123), [SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md](/Users/liubu/hx/hxWork/35.AI测试/claw/smartclaw/docs/SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md#L630)
   - 风险：实现时要么 orchestrator 被迫自行“补脑”二次推断，要么 planner 输出不断扩张并散落到代码里，最终造成职责边界重新混乱。

3. Artifact 只定义了“引用 + 文件存储”，但没有定义最小标准化字段和校验规则，难以成为跨业务稳定总线。
   - 证据：文档强调 planner 要依赖 “ready artifacts 摘要” 和结构化结果持续 replanning，但当前 artifact 示例只有 `type / producer_step / status / path` 与任意 JSON 内容，没有规定 `schema_version`、标准摘要、校验结果、来源上下文、消费接口。见 [SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md](/Users/liubu/hx/hxWork/35.AI测试/claw/smartclaw/docs/SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md#L134), [SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md](/Users/liubu/hx/hxWork/35.AI测试/claw/smartclaw/docs/SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md#L295), [SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md](/Users/liubu/hx/hxWork/35.AI测试/claw/smartclaw/docs/SMARTCLAW_UNIFIED_DYNAMIC_ORCHESTRATION_DESIGN.md#L607)
   - 风险：不同业务 step 产物格式各自为政，planner 最终仍要消费松散文本摘要，无法真正支撑“少改核心代码、主要靠能力注册扩展”。

## 修改建议

1. 在不把 Step Registry 做重的前提下，补一个“最小可执行契约”：
   - 增加 `required_inputs`
   - 增加 `input_sources` 或允许的 artifact type
   - 增加 `completion_signal`
   - 增加 `side_effect_level`
   这样仍然不是 workflow engine，但 planner 至少不再纯靠猜。

2. 把 planner 输出从“候选 todo 列表”升级为“可执行 todo plan”：
   - 至少补 `step_id`
   - 补 `resolved_inputs`
   - 补 `consumes_artifacts`
   - 补 `execution_mode`
   - 补 `approval_required`
   - 补 `plan_version` 或 schema version

3. 给 Artifact 定义统一最小外壳：
   - `artifact_id`
   - `artifact_type`
   - `schema_version`
   - `status`
   - `summary`
   - `producer_step`
   - `validation`
   - `payload_path`
   Phase 1 仍可把正文存文件，但壳层最好统一。

4. 明确规则优先级：
   - Step Registry 描述“天然属性”
   - Capability Pack 描述“业务域允许范围与默认偏好”
   - governance middleware 负责“运行时强制执行”
   并写出冲突覆盖顺序，避免后续实现分叉。

5. 先冻结一版统一运行时 stage：
   - 例如 `plan -> dispatch -> execute -> normalize -> review -> synthesize -> finish`
   然后所有 middleware、状态机、日志、追踪都引用这一套 stage 名字，不再混用。

6. 给 replanning 增加全局 guardrail：
   - session 最大 replanning 次数
   - 同一 step 最大失败次数
   - 同类错误连续出现时直接升级为人工确认
   - 超预算后返回“当前可交付结果 + 阻塞原因”

7. 补一段“Phase 1 必须先定死的四个 schema”：
   - StepDefinition
   - TodoPlan
   - ArtifactEnvelope
   - StepRunRecord
   这样后续开发不会边写边发明协议。

## 风险/待确认点

1. 这份设计适合作为总方向文档，但还不够直接指导编码；建议在开工前再补一份“Phase 1 运行时契约设计”，专门写清楚四个核心 schema 和状态流转。
2. 文档当前最强的部分是分层和方向判断，最弱的部分是运行时协议。如果协议层不先补，最终实现很可能退回到当前关键词 planner 的升级版，而不是可扩展的通用动态规划框架。
3. 若近期目标是快速打通试点，建议先以安全治理场景落地，但在试点前仍要先冻结最小协议；否则试点代码会直接变成安全场景特化逻辑，反而污染核心框架。
